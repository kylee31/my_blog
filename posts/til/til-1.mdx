### 좋은 Readme 작성법

- 각 프로젝트 별 Readme를 작성해두었지만 일반 Readme가 아닌 포트폴리오에 가까운 내용을 작성해두었다는 점을 깨달았다. 좋은 Readme를 작성하기 위해 [infograb-좋은 Readme 작성법](https://insight.infograb.net/blog/2023/08/23/good-readme/) 참고, 실제 오픈소스 라이브러리의 Readme(쿠버네티스 등)를 참고하여 수정해야겠다고 느꼈다.

- ![](https://velog.velcdn.com/images/kylee31/post/d44149b7-afaf-46d4-9de3-3154479e7c52/image.png)

1. 주요 컴포넌트, 실제 실행 이미지 넣기
2. 호스팅 주소 삽입? 프로젝트 매핑하기
3. as-is, to-be 문서로 작성해서 최종 프로젝트를 위한 docs를 따로 빼서 넣기 (gitbook)

들어가면 좋은 내용: 주차별 요구사항, 트러블 슈팅, 설계 관련 고민들, 하고 싶은 얘기

- 꼭 보여주고 싶은 내용을 강조하기 (모든 내용을 꾸밀 필요 없음)

- draw.io
- best-readme-template
- docs(gitbook)

- readme가 아깝지 않은 내용이 들어가야 함! 상세 설명은 docs로 첨부

- 브랜치 전략, 깃 컨벤션은 유난히(똑같은 커밋 메시지 100개 떄리는 등) 못하지만 않으면 된다.. 회사마다 기준이 다 다르기 때문
  못해서 눈에 띄지 않는 정도, 시간을 많이 쓰지 않으면 된다.
- storybook 사용한거 걸어놓는게 좋은듯?(프론트엔드) 컴포넌트 나열한 페이지 짜보는 것도 좋음
- package.json 정리하는게 프로젝트가 예뻐보임

<hr />

👩‍💻 트러블 슈팅과 관련지어 프로젝트 정리하면서 기술 스택을 **왜** 사용했는지, 선정 이유는 **무엇**인지, **어떻게** 해결했는지
잘 설명하는 것이 어려운 일이고 중요한 일임을 다시 한번 느꼈다.
