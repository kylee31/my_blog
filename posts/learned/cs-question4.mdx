**질문1.** 클라이언트 사이드 렌더링(CSR)과 서버 사이드 렌더링(SSR)의 개념과 장/단점을 설명해주세요.

**답변1.** 먼저 CSR은, 클라이언트 측에서 렌더링 하는 방식입니다. 초기 로딩 시 빈 페이지 또는 미리 정의된 레이아웃만을 서버에서 받습니다.
페이지 데이터는 JavaScript를 사용하여 비동기적으로 서버로부터 가져와 동적으로 렌더링됩니다.
SSR은 웹 서버에서 페이지의 HTML 및 컨텐츠를 동적으로 생성하여 클라이언트에 제공하는 방식입니다. 해당 방식은 서버 측에서
HTML이 렌더링되므로 SEO 최적화에 효과적이지만, 매번 서버 측에서 렌더링이 발생하므로 서버 부하가 커질 수 있다는
단점도 존재합니다.

- SSR은 CSR보다 화면 출력이 매끄럽지 못할 수 있다.
- CSR은 초기 로딩 속도가 느리지만 이후 동작이 클라이언트 측에서 발생하므로 빠른 로딩이 필요할 때 사용, SSR은 SEO 최적화가
  필요할 때 사용하는 것이 적합합니다.
- ❗ SPA 개념과 CSR의 혼동이 있었음. 주의하기

**[Keyword]** 트레이드 오프, 렌더링, 비동기, 유저 사용성

**질문2.** TCP/UDP에 대해서 설명해주세요.

**답변2.** 두가지는 네트워크에서 데이터를 보내기 위해 사용하는 프로토콜입니다. TCP는 신뢰성을 보장하고,
연결 지향적인 통신 방식이며, UDP보다 속도가 느립니다. UDP는 비연속적인 통신 방식이며 오류 발생 시 데이터
재전송이 이뤄지지 않기 때문에 신뢰성이 떨어지는 것으로 알고 있습니다. HTTP1과 2에서는 TCP를 전송에 사용하지만,
HTTP3부터는 UDP를 사용하고 있으며, 현재 구글과 네이버가 HTTP3를 도입한 것으로 알고 있습니다.
따라서 신뢰성이 필요한 곳에서는 TCP, 실시간성과 빠른 연결이 필요하면 UDP 사용할 수 있습니다.

- TCP는 연결을 시작하기 전에 '3-way handshake'를 통해 두 시스템이 서로에 대해 알아보고, 데이터 전송이 끝나면 '4-way handshake'로 연결을 종료합니다.
- UDP는 handshake 과정이 없어서 연결 설정이 필요 없고, 이로 인해 더 빠르게 데이터를 보낼 수 있습니다.
- HTTP1,2는 TCP 사용, HTTP3(QUIC)는 UDP 사용. 구글 네이버가 HTTP3 도입

**[Keyword]** 3-way handshake, 4-way handshake, 신뢰성, 데이터 손실, 속도

**질문3.** http와 https의 차이점에 대해 설명해주세요

**답변3.** 두가지 모두 웹 통신 규약입니다. http는 보안 설정이 되지 않는 웹 프로토콜이기 때문에 따라서 웹 사이트 공격에 취약합니다.
https는 보안 설정이 되어있는 웹 프로토콜입니다. SSL(보안 소켓 계층)/TLS 프로토콜을 사용하여 데이터를 암호화 하므로 http보다
안정성이 높고 기밀성, 무결성을 보장할 수 있습니다. Nginx와 같은 웹 서버를 통해 인증서를 발급받아 웹 사이트에
보안 설정을 할 수 있습니다.

- 기밀성 : 허가받은 사람만 접근 가능
- 무결성 : 정보가 원래 내용대로 유지

**[Keyword]** SSL, TCP/UDP, 인증서

**질문4.** VanillaJS와 비교하여 리액트를 사용하는 이유에 대해 설명해주실 수 있을까요?

**답변4.** 가장 큰 이유를 설명하자면, 리액트는 가상 돔이라는 개념을 사용하고 있어 변경이 일어난 트리 노드에서만 렌더링이
발생합니다. 즉, 이러한 동작은 모든 돔 노드가 리렌더링되는 바닐라 자바스크립트보다 가볍고 불필요한 동작을 줄일 수 있다는
장점이 존재합니다. 이 외에도 자바스크립트 내 HTML 태그 작성이 가능한 JSX 문법으로 가독성이 좋은 코드를 작성할 수 있다는
장점이라고 생각합니다. JSX 문법으로 컴포넌트화하여 상태 관리가 용이해지며, 코드 재사용성과 유지보수, 가독성이 향상됩니다.

**[Keyword]** 상태 관리, JSX

**질문5.** 상태의 불변성이 중요한 이유가 무엇인가요?

**답변5.** 리액트에서 상태 불변성이 중요한 이유는, 렌더링과 관련되어 있기 때문입니다. 상태 변화가 발생하면 리액트는 가상
돔의 이전 상태 값을 비교하여 변경이 일어난 노드에 대해 실제 돔에 변경 값을 반영하게 됩니다. 따라서 이전 상태 값이
변경되지 않아야 변경된 값을 비교하는 동작을 통해 렌더링을 일으킬 수 있기 때문입니다. 비교는 깊은 비교가 아닌 메모리 주소를
비교하는 얕은 비교가 일어납니다. (모든 값을 다 깊은 비교하는 건 비효율적)

- 상태 불변성을 유지하는 건 안전성, 성능 측면에서 이점이 크다.

**[Keyword]** 깊은/얕은 비교, 변경과 렌더링

**질문6.** react 클래스형과 함수형의 차이를 설명해주세요. 어떤 방식을 주로 사용하였고 그 이유가 뭔지 답변해주세요.

**답변6.** 클래스 형식과 함수 형식으로 컴포넌트를 선언하고 사용하는 방식에서 차이가 있습니다. 클래스 형식에서는 생명 주기
메서드를 활용하여 this 키워드를 통해 상태 값을 다루지만, 함수 형식은 생명 주기 메서드가 없기 때문에 이를 대신하여
Hook이라는 형태로 제공되는 여러 함수를 사용하여 생명 주기 기능과 상태 관리를 수 있습니다. React 초기에는 클래스형이
사용되었으나, 현재는 Hook이 다양해지고 여러 기능도 많아지며 클래스 형식이 레거시가 되었고, React 또한 함수형을 권장하고
있으며 사용법이 훨씬 간단하고 편리하기 때문에 함수형을 사용하고 있습니다.

**[Keyword]** 컴포넌트 생명 주기, 메모리 할당
