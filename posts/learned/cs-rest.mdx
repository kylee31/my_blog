> HTTP와 REST가 무엇인지, REST API와 RESTful API 개념 문서로 정리해두기

### Web의 4가지 요소

HTTP, REST 등을 알아보기 전 Web에 대해 정리해보자. Tim Berners-Lee가 인터넷 상의 하이퍼텍스트 시스템을 만들기 위해 작성한
제안서로, 1990년 월드 와이드 웹(WWW)이라는 이름으로 세상에 나왔다. 이러한 웹은 크게 4가지 요소로 이루어져있다.

1. HTML : 하이퍼텍스트 문서를 표현하기 위한 텍스트 형식의 하이퍼텍스트 마크업 언어(웹페이지를 만드는 컴퓨터 언어)
2. URI/URL : 원하는 웹페이지에 방문할 수 있도록 도와주는 주소 체계 (URI의 하위 개념, URL/URN)
3. Web Browser/Web Server : 웹 페이지를 주고받는 소프트웨어
4. HTTP : 웹 브라우저와 웹 서버가 통신할 때 사용하는 통신 규칙

처음 등장한 HTTP는 매우 단순한 규칙이었다. 그러나 웹이 성장하면서 기존 HTTP 기능만으로는 성능, 보안, 안정성을 확보하는게
어려워졌다. 이러한 문제를 해결하기 위해 HTTP는 보다 풍부한 기능을 가진 프로토콜로 발전한다. 오늘날에는 단지 HTML과 같은
텍스트 파일 뿐 아니라 이미지 오디오 동영상 같은 다양한 멀티미디어 파일을 전송하는 매우 중요한 프로토콜로 군림하고 있다.
인터넷에서 가장 중요한 통신 규약이 된 HTTP는 웹을 넘어서 인터넷이 동작하는 근간이라고도 할 수 있다.

이제 아래에서 HTTP, REST를 알아보자.

### HTTP

- HTTP(Hypertext Transfer Protocol)란, 단순히 정리하자면 정보를 주고 받을 수 있는 통신 규약이다.

HTTP는 인터넷(WWW, 월드 와이드 웹) 상에서 클라이언트와 서버 간 이루어지는 의사소통이다. request header와 response header를 통해 이루어지며,
주로 html을 주고 받는다. HTML, CSS, JS, 이미지와 같은 파일들이 서로가 주고받는 컨텐츠라면 이러한 컨텐츠를 주고 받기
위해 서버와 클라이언트가 이해할 수 있는 공통의 약속(메세지)이 필요한데, 그 메세지를 HTTP라고 하는 것이다.
HTTP는 크게 Request와 Response를 위한 메세지로 구분되어 있다고 볼 수 있다.

✔ **HTTP 메시지**

요청 메시지와 응답 메시지로 구성되며, 요청 메시지에 포함된 메소드에 따라 서버는 해당 작업을 수행하고 응답 메시지를 반환한다.

**1. 요청 메시지 (Request Message)**

- HTTP 메소드(HTTP Request Method) : 요청 메시지의 요청 줄에 포함되어 클라이언트가 서버에 원하는 작업을 명시한다.<br/>
  ex) GET, POST, PUT, PATCH, DELETE

**2. 응답 메시지 (Response Message)**

- 상태 코드 : 요청에 대한 응답 상태를 알려주는 코드

### REST

- REST란, Representation State Transfer의 약자로 쉽게 말해 자원 상태 전달이라고 할 수 있다. REST는 웹과 같은 인터넷
  규모 분산 하이퍼미디어 시스템 의 아키텍처가 어떻게 작동해야 하는지에 대한 일련의 제약 조건을 정의한다.

REST는 자원을 이름(자원의 표현)으로 구분해 해당 자원의 상태(정보)를 주고받는 모든 것을 의미한다. 자원(resource)의
표현(representation)에 의한 상태 전달을 뜻합니다.

▶ REST는 웹의 장점을 최대한 활용할 수 있는 아키텍처 스타일로, 웹의 기존 기술과 HTTP 프로토콜을 그대로 활용한다.

- **REST 구성요소**

  1. 자원(Resource): URI <br/>
     HTTP URI(Uniform Resource Identifier)를 통해 자원(Resource) 명시

  2. 행위(Verb): HTTP Method <br/>
     HTTP Method(POST, GET, PUT, DELETE)를 통해 해당 자원에 대한 CRUD Operation 적용
     ❓ CRUD Operation <br/>
     ( Create : 생성(POST), Read : 조회(GET), Update : 수정(PUT), Delete : 삭제(DELETE), HEAD: header 정보 조회(HEAD) )

  3. 표현(Representation of Resource) <br/>
     Client가 자원의 상태(정보)에 대한 조작을 요청하면 Server는 이에 적절한 응답(Representation)을 보낸다.
     REST에서 하나의 자원은 JSON, XML, TEXT, RSS 등 여러 형태로 표현할 수 있다. 보통, JSON 혹은 XML를 통해 데이터를
     주고받는 것이 일반적이다.

- **REST 특징**

  1. Server-Client(서버-클라이언트 구조)

  2. Stateless(무상태)
     HTTP 프로토콜은 Stateless Protocol이므로 REST 역시 무상태성을 갖는다.

  3. Cacheable(캐시 처리 가능)

  4. Layered System(계층화)

  5. Code-On-Demand(optional)
     Server로부터 스크립트를 받아서 Client에서 실행한다. 반드시 충족할 필요는 없다.

  6. Uniform Interface(인터페이스 일관성)

✅ **REST API 설계 원칙**

REST API란 REST 기반으로 서비스 API를 구현한 것으로, REST의 원리를 따르는 API를 RESTful API라고 한다.

REST API 설계 시 가장 중요한 원칙은 자원(Resource) 중심의 설계이다. 각 자원은 고유한 URI를 가지며, 자원에 대한 행위는
HTTP 메소드를 통해 표현된다. 또한, 상태가 없는(Stateless) 통신을 지향하며, 각 요청은 독립적으로 처리되어야 한다.
이외에도 캐시 가능한(Cacheable) 응답을 사용하여 효율성을 높이는 것도 중요한 설계 원칙이라고 할 수 있다.
